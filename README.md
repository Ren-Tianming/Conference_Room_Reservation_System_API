# 会議室予約システム V2（エンタープライズ向け改良版）

本プロジェクトは、既存の V1 CRUD ベースの会議室予約システムを土台として、実務で求められやすい認証・キャッシュ・排他制御・責務分離を追加した、バックエンド志向の V2 構成です。

主な強化ポイントは以下のとおりです。

- ユーザー登録 / ログイン / JWT 認証
- Access Token + Refresh Token
- パスワードハッシュ化（Argon2）
- Redis キャッシュ（会議室一覧・日次予約一覧）
- Redis によるトークン管理（Refresh Token 保存、Access Token ブラックリスト）
- 予約重複チェック
- 同時予約対策（Redis 分散ロック + データベース行ロック）
- 管理者による会議室作成・保守
- `/health` / `/ready` のヘルスチェック用エンドポイント
- レイヤ分離された構成（api / core / db / models / schemas / services）

---

## 1. なぜ V1 の構成のままでは厳しいのか

V1 は CRUD 学習用としては十分ですが、そのまま機能追加を続けると保守性が急激に落ちます。

主な課題は以下のとおりです。

- `main.py` にルーティングが集中しやすく、機能追加時に肥大化しやすい
- `crud.py` に単純な DB 操作と業務ロジックが混在しやすい
- `schemas.py` / `models.py` が最小構成のままで、認証・権限・拡張に弱い
- パスワードハッシュ化、認証、認可が未整備
- 予約重複判定はあっても、同時アクセス時の競合対策が不足
- キャッシュ層やトークンのライフサイクル管理がない

そのため、V2 では単なる機能追加ではなく、実務に近いバックエンド構成へ再設計しています。

## 2. 起動方法

```bash
cp .env.example .env
docker compose up
```

起動後、以下にアクセスできます。

- Swagger UI: `http://localhost:8000/docs`
- OpenAPI JSON: `http://localhost:8000/openapi.json`

---

## 3. 想定する基本的な利用手順

1. `POST /api/v1/auth/register` でユーザー登録
2. `POST /api/v1/auth/login` でログイン
3. 返却された `access_token` を使って保護対象 API を呼び出す
4. `POST /api/v1/auth/refresh` でトークンを更新する
5. 管理者が `POST /api/v1/rooms` で会議室を登録する
6. 一般ユーザーが `POST /api/v1/bookings` で予約を作成する

---

## 4. 現在の同時予約対策

V2 では、予約競合に対して二重の保護を行っています。

### 4.1 Redis ロック
同一会議室への同時予約リクエストを一時的に直列化し、瞬間的な競合を抑制します。

### 4.2 データベース行ロック
予約作成時に対象会議室のレコードへロックをかけたうえで、再度時間帯の重複チェックを行います。

これにより、V1 のような「先に検索して、あとで登録するだけ」の実装よりも、競合に強い構成になっています。

---

## 5. 今後の拡張候補

より本番運用に近づけるために、次の機能追加が有効です。

- Alembic によるマイグレーション管理
- Nginx + Gunicorn/Uvicorn を用いた複数インスタンス構成
- PostgreSQL の Exclusion Constraint による時間帯重複の DB レベル保証
- pytest による自動テスト
- より細かい RBAC（ロールベースアクセス制御）
- 操作履歴 / 監査ログ
- レート制限、防刷対策、冪等キー
- GitHub Actions などによる CI/CD

---

## 6. V1 との位置づけの違い

この V2 は、V1 に対して単純にコードを継ぎ足したものではありません。

- V1: CRUD 学習用の最小構成
- V2: 実務を意識したバックエンド構成

そのため、GitHub ポートフォリオとして見せる際にも、V1 よりはるかに「実案件に近い設計意図」が伝わりやすくなります。

## 7. このプロジェクトで学べること

この構成を通して、以下のようなバックエンド実務の基礎を学べます。

- FastAPI による API 設計
- SQLAlchemy を用いた ORM 設計
- Pydantic によるバリデーション
- JWT ベースの認証基盤
- Redis を用いたキャッシュ戦略とトークン管理
- 予約システムに必要な競合制御の考え方
- レイヤ分離による保守しやすいプロジェクト構成
